# Wayfile 1.0.0 — W1 Token Adoption Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Re-point Wayfile's signature `bifrost` theme tokens (and the fixed file-type tints) to the new handoff's lighter steel-grey obsidian ladder + softer tan-gold `#D4AA6A`, so the whole app re-tints to the new look with no structural changes.

**Architecture:** Values-only re-skin. The theme TOML (`themes/bifrost.toml`) is the source of every colour the QML reads via the `theme` context property (`ThemeLoader`). `Theme.qml` is a singleton that aliases those tokens and auto-derives `goldGlow/goldWash/goldLine` from `gold` (so they re-tint for free). `FileTypeColors.qml` holds the fixed, non-theme-overridable file-type tints. We change the **values** of existing tokens — never the keys, never component structure.

**Tech Stack:** Qt6/QML, TOML themes (`third_party/toml.hpp`), Qt6::Test (`tst_themeloader`), `cmake`/`ctest`.

## Context an implementer needs (zero-assumption brief)

- **Where colours live:** `themes/bifrost.toml` `[colors]` table → parsed by `ThemeLoader` (`src/services/themeloader.{h,cpp}`) → exposed to QML as the `theme` context property → aliased in the `Theme` QML singleton (`src/qml/theme/Theme.qml`) → read by components as `Theme.<token>`.
- **`Theme.qml` derivations:** `goldLine/goldGlow/goldWash` are computed from `Theme.gold` at runtime (lines ~46-48). Because `gold` becomes `#D4AA6A` via the TOML, these glow helpers re-tint automatically — **no Theme.qml colour edits are needed in W1.** (The W2 workstream later moves ramp derivation into Theme.qml; not now.)
- **Two radius systems (do not confuse):**
  1. **Config-driven, user-facing, dominant:** `Theme.radiusSmall/Medium/Large` (← `config.*`). Used ~108×. These are the *visible* corner radii. **W1 does NOT touch these** (they are user preferences with their own defaults in `ConfigManager`).
  2. **Fixed handoff radii:** `Theme.radiusSm/Md/Lg/Tab/Button/Pill/Tile/Row` (hardcoded in `Theme.qml`). Largely vestigial — the Quill bridge in `Main.qml:240-242` maps Quill's `radiusSm/radius/radiusLg` to the *config* radii, and `radiusMd`/`radiusLg`/`radiusTile` have **no native consumers**. Only `radiusSm` (3 native consumers: `ContextMenuRow`, `ContextMenuSubmenuRow`, `Main.qml`) has any real effect. W1 aligns the `sm/md/lg` trio to the handoff for correctness; impact is minimal.
- **`s_defaults` (fallback palette in `themeloader.cpp`):** the hardcoded fallback used when a theme omits a token. It is **separate** from `bifrost.toml` and is **NOT changed in W1** (changing it would break the default-fallback tests `testFallbackForMissingColors`, `testMissingThemeFile`, `testNewTokensFallBackToDefaults`, `testInvalidToml`, `testEmptyThemeFile`, `testColorsSectionMissing`). Because `bifrost.toml` provides every token, defaults never surface for bifrost.
- **The TDD lever:** `tests/tst_themeloader.cpp::testBifrostObsidianGoldTokens()` already asserts bifrost's **current** values. Re-tinting `bifrost.toml` makes it fail — that is our RED. Update the assertions first, watch them fail, then re-tint to GREEN.
- **catppuccin themes are out of scope for W1** — they are NOT dead (they are the `ConfigManager` default fallback, the dark/light toggle target in `SettingsSectionLook.qml`, and asserted by `tst_themeloader`/`tst_configmanager`). Their removal belongs to W2 (theming).
